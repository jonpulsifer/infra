- NFS/PXE boot server — **boot-critical for [[Hosts/rackpi5]]** ([[ADR/0008 Diskless netboot for rackpi5]]); monitored by folly.
- Only Pi with NVMe storage (128 GB Patriot P300). Config: `nix/hosts/spore.nix`.
- Reached as `spore.lolwtf.ca`; the Tailscale management UI uses `spore.pirate-musical.ts.net/spore`.
- ## Spore application
	- The read-only iPXE catalog/status app runs directly as `spore.service`; there is no Spore container. Node listens only on loopback and nginx owns network access.
	- Desired host/profile/script policy is generated by Nix. Host MACs come from `terraform/network/unifi/folly/clients.yaml`; policy lives in `nix/hosts/spore.nix`. Change either through a PR, then rebuild the host from merged `main`.
	- PXE clients may read only `/spore/api/boot/…` and `/spore/api/scripts/…` through the existing default PXE vhost. The management UI uses a separate Tailscale-authenticated vhost with `spore.pirate-musical.ts.net` as a direct MagicDNS alias; `spore.lolwtf.ca` remains its canonical server name.
	- Runtime observations live in `/var/lib/spore/observations.db`. The database is not authoritative for boot behavior; losing it resets history only. The NixOS backup timer writes retained SQLite backups under `/var/backup/spore`.
	- `spore.service` has no `Requires=`, `PartOf=`, or restart coupling to dnsmasq, nginx, NFS, or the static PXE root. A web-app failure is alerted separately and must not prevent static boot assets from being served.
- ## Build and recovery
	- Validate without activating: `nix build .#nixosConfigurations.spore.config.system.build.toplevel --no-link` on a native aarch64 builder.
	- Deploy from merged Git with the standard NixOS host workflow. Roll back with `sudo nixos-rebuild switch --rollback`; static PXE/NFS services remain independently managed throughout.
	- On first migration from the legacy mixed-state database, Spore preserves a recoverable copy before creating the observations-only schema. Restore that backup only for forensic recovery; reviewed Git/Nix catalog state remains authoritative.
	- The pre-start migration is best-effort. If observation storage is corrupt or unavailable, `spore.service` still starts with catalog-backed boot decisions while health remains degraded and observation writes are disabled.
